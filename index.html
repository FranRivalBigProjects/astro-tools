How Astro partial hydration works

Partial hydration means only specific components get JavaScript — the rest of the page is zero-JS static HTML. Astro calls these interactive components Islands. Every non-Island component ships as plain HTML with no client-side overhead.

--- // src/pages/index.astro import Header from ’./Header.astro’ // Zero JS — static HTML import HeroSection from ’./HeroSection.astro’ // Zero JS — static HTML import SearchBar from ’./SearchBar.jsx’ // Island — needs JS import ProductGrid from ’./ProductGrid.jsx’ // Island — needs JS import Comments from ’./Comments.jsx’ // Island — needs JS import Footer from ’./Footer.astro’ // Zero JS — static HTML —

Why zero-JS by default matters
Traditional React/Vue/Angular apps hydrate the entire component tree — even components that never change (a static nav bar, a footer, article text). Astro inverts this: the default is no JavaScript. You opt-in specific components to receive JS. A typical content site built with Astro ships 0KB of JS for the reader until they interact with an Island. This directly impacts LCP (Largest Contentful Paint), TTI (Time to Interactive), and Core Web Vitals scores.

client:visible — the most impactful directive
client:visible uses IntersectionObserver to wait until the component enters the viewport before fetching and executing its JavaScript. For a page with a carousel at the bottom, this means the carousel's JS bundle is never loaded for users who don't scroll that far. In high-traffic scenarios, this can reduce JavaScript transfer by 40-70% compared to eager loading everything.

Framework agnostic — React, Vue, Svelte in the same page
Each Astro Island can use a different framework. client:only="react", client:only="vue", and client:load on a Svelte component can all exist on the same page. Each Island loads only its own framework runtime — so a page with one React Island ships ~40KB of React runtime, not the full React app bundle. A pure-HTML page ships nothing.

FAQ — Astro partial hydration
Astro partial hydration sends JavaScript only to specific interactive components called Islands. The rest of the page is static HTML — zero JavaScript. A standard blog post page in Astro might have a React search bar (Island) and a Vue comment section (Island), while the article content, header, and footer ship as plain HTML. This drastically reduces JS bundle size and improves performance compared to SPAs that hydrate everything.
client:load — JavaScript is fetched and the component hydrates immediately when the page loads. Use for components visible above the fold that need instant interactivity (search bars, interactive navigation). client:visible — hydration is deferred until the component scrolls into the viewport using IntersectionObserver. Use for below-fold components. A user who never scrolls down never loads that component's JS at all. This is the most impactful directive for performance optimization.
client:only skips server-side rendering entirely — the component has no HTML output until it hydrates in the browser. Use it when: (1) The component depends on browser APIs unavailable during SSR (window, document, localStorage). (2) A third-party widget doesn't support SSR. (3) You're using a framework where SSR is problematic. You must specify the framework: client:only="react". The downside: no HTML is present before JS loads, which can hurt LCP and cause layout shift.
Both reduce client JavaScript, but differently. Astro Islands are framework-agnostic — you can mix React, Vue, Svelte in one page. Non-Island Astro components are pure HTML with zero runtime. React Server Components stay within the React ecosystem and allow fine-grained server/client separation within a component tree. Astro is a better choice when you want framework flexibility or are building primarily content-driven sites. RSC is better when you're committed to React and want deep integration with React's rendering model.
